## Project Overview

This project simulates a real-world data engineering workflow for an e-commerce business, covering:

| Stage | Description | Script |
|-------|-------------|--------|
| **Ingestion** | Pull data from APIs & generate synthetic records; load to PostgreSQL staging | `ingestion.py` |
| **Transformation** | Clean, validate, enrich & derive new columns | `transformation.py` |
| **Modeling** | Build star-schema DW with dimensions & fact tables | `modeling.py` |
| **Analytics** | Run 13 analytical queries; export Markdown report | `stats.py` |

**Database:** PostgreSQL (two schemas — `staging` and `dw`)

## Data Sources

### 1. REST Countries API (Public — No Key Required)
- **URL:** `https://restcountries.com/v3.1/all`
- **Fields ingested:** country code, name, region, subregion, population, currency
- **Usage:** Populates `staging.countries` → `dw.dim_location`

### 2. Open Exchange Rates API *(Optional)*
- **URL:** `https://openexchangerates.org/api/latest.json`
- **Key required:** Free tier available at openexchangerates.org
- **Fallback:** Static rates are hard-coded if no API key is provided
- **Usage:** Populates `staging.currencies`

### 3. Synthetic E-Commerce Data (Faker + NumPy)
Controlled by `config/api_config.json`:

| Setting | Default | Description |
|---------|---------|-------------|
| `faker_seed` | 42 | Reproducible random seed |
| `num_customers` | 500 | Customer records to generate |
| `num_products` | 200 | Product SKUs to generate |
| `num_orders` | 5000 | Order transactions |
| `date_range.start` | 2022-01-01 | Order date range start |
| `date_range.end` | 2024-12-31 | Order date range end |

**Product categories:** Electronics, Clothing, Home & Garden, Books, Sports, Beauty, Food & Drink, Toys

## Star Schema Design

### Dimension Tables

#### `dw.dim_date`
Full calendar dimension from 2021–2026 (~2,192 rows)

| Column | Type | Description |
|--------|------|-------------|
| `date_key` | INTEGER PK | YYYYMMDD surrogate key |
| `full_date` | DATE | Calendar date |
| `day_of_week` | SMALLINT | 1=Mon … 7=Sun |
| `day_name` | VARCHAR | "Monday" … "Sunday" |
| `week_of_year` | SMALLINT | ISO week number |
| `month_number` | SMALLINT | 1–12 |
| `month_name` | VARCHAR | "January" … |
| `quarter` | SMALLINT | 1–4 |
| `year` | SMALLINT | Gregorian year |
| `is_weekend` | BOOLEAN | True for Sat/Sun |
| `season` | VARCHAR | Winter/Spring/Summer/Autumn |

#### `dw.dim_location`
Country-level geography (one row per country)

#### `dw.dim_customer`
Customer SCD Type 1 dimension with derived attributes (age, age_group, loyalty_tier)

#### `dw.dim_product`
Product dimension with category hierarchy: Category → Subcategory → Product

#### `dw.dim_payment`
Payment method lookup (Credit Card, PayPal, Bank Transfer, Crypto, BNPL)

### Fact Tables

#### `dw.fact_sales` *(Line-Item Grain)*
One row per order line item — the most granular fact table.

**Key measures:** `quantity`, `unit_price`, `discount_amt`, `line_gross`, `line_net`, `line_cost`, `line_margin`, `margin_pct`, `shipping_cost`, `days_to_ship`

#### `dw.fact_orders` *(Order Grain)*
One row per order header — pre-aggregated for faster order-level analytics.

**Key measures:** `num_line_items`, `total_quantity`, `gross_revenue`, `total_discount`, `net_revenue`, `total_cost`, `gross_margin`, `shipping_cost`, `days_to_ship`

### Schema Diagram

```
                        dim_date
                           │
                    order_date_key
                           │
dim_payment ──payment_key──┤              dim_product
                           │             /
                     fact_sales──product_key
                     fact_orders
                           │ \
              customer_key │  location_key
                           │              \
                    dim_customer        dim_location
```

## Setup & Installation

### Prerequisites
- Python 3.10+
- PostgreSQL 14+ (running locally or remote)
- Git

### 1. Clone the repository
```bash
git clone https://github.com/YOUR_USERNAME/data-warehousing-project.git
cd data-warehousing-project
```

### 2. Create and activate a virtual environment
```bash
python -m venv venv
# macOS/Linux:
source venv/bin/activate
# Windows:
venv\Scripts\activate
```

### 3. Install dependencies
```bash
pip install -r requirements.txt
```

### 4. Create the PostgreSQL database
```sql
-- In psql or pgAdmin:
CREATE DATABASE ecommerce_dw;
```

### 5. Configure database connection
Edit `config/db_config.json`:
```json
{
  "host": "localhost",
  "port": 5432,
  "database": "ecommerce_dw",
  "user": "postgres",
  "password": "YOUR_PASSWORD",
  "staging_schema": "staging",
  "dw_schema": "dw"
}
```

### 6. (Optional) Add Open Exchange Rates API key
Edit `config/api_config.json` and set `"app_id"` to your key.  
The pipeline works without it using static fallback rates.

## Script Reference

### `scripts/ingestion.py`
- Calls the REST Countries API (`restcountries.com`)
- Calls the Open Exchange Rates API (or uses fallback)
- Uses `Faker` to generate 500 customers, 200 products, 5,000 orders, ~12,000 order items
- Saves all raw data to `data/raw/*.csv`
- Creates the `staging` schema and all tables
- Loads every table into PostgreSQL staging

### `scripts/transformation.py`
**Transformations applied per entity:**

| Entity | Transformations |
|--------|----------------|
| Countries | Deduplicate, title-case names, fill nulls, validate code length |
| Currencies | Deduplicate, normalise codes, cap extreme outliers (IQR), validate date |
| Products | Deduplicate by SKU, validate prices (cost ≤ price), IQR outlier capping, derive `gross_margin_pct` |
| Customers | Deduplicate by email, validate email regex, derive `full_name`, `age`, `age_group`, remove impossible ages |
| Orders | Parse dates, remove orphan orders, derive `order_year/quarter/month/week/dow`, compute `days_to_ship` |
| Order Items | Validate FKs, remove zero-price rows, derive `line_gross`, `line_net`, `line_cost`, `line_margin`, `margin_pct` |

### `scripts/modeling.py`
- Drops and recreates the `dw` schema
- Builds a full calendar `dim_date` (2021–2026)
- Loads all 5 dimension tables with surrogate keys
- Builds `fact_sales` (line-item grain) from `staging.order_items` joined to orders
- Builds `fact_orders` (order grain) by aggregating items to order level
- Creates all performance indexes
- Exports DDL to `sql/dw_schema.sql`

### `scripts/stats.py`
Runs 13 analytical queries against the DW and generates `reports/stats_report.md`:
1. Overall KPI summary
2. Revenue by year (YoY growth)
3. Quarterly revenue trend
4. Top 10 products
5. Revenue by category
6. Customer segmentation by loyalty tier
7. Top 10 countries by revenue
8. Revenue by region
9. Payment method analysis
10. Order status breakdown
11. Shipping performance
12. Revenue by customer age group
13. Gross margin by category

## Key Insights

*(Based on default dataset of 500 customers / 200 products / 5,000 orders)*

- **Electronics** is the top revenue-generating category due to high average unit price
- **Platinum** loyalty tier customers spend ~3–4× more per order than Bronze customers
- **Americas** and **Europe** together account for over 70% of total revenue
- **Credit Card** is the preferred payment method (~33% of orders)
- Over **65% of orders** are delivered within 5 days (express/standard shipping)
- Order cancellation and return rate remains below **10%**
- Year-over-year revenue growth is visible from 2022 → 2024

See `reports/stats_report.md` for the full generated report after running the pipeline.

## Design Decisions

**Why a Star Schema?**
Star schemas provide the best query performance for analytical workloads. The denormalized structure reduces JOIN complexity and allows BI tools to generate simple, fast queries.

**Why two fact tables?**
`fact_sales` (line-item) enables product-level analytics; `fact_orders` (order) enables customer-level analytics. Pre-aggregating at order level avoids expensive GROUP BY on the largest table for common queries.

**Why Faker for synthetic data?**
Using a seeded random generator ensures reproducibility — any team member running the pipeline gets the identical dataset, making collaboration and testing deterministic.

**Why keep `staging` and `dw` as separate schemas?**
Separation of concerns: staging holds raw/enriched landing data, DW holds the modelled star schema. This allows re-running transformation or modeling independently without re-ingesting.
